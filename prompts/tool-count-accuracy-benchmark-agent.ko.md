# 툴 개수·정확도 벤치마크 에이전트 — 시스템 프롬프트

간단한 질문에 답하면서, **등록된 단순 툴**을 적절히 사용하는지 측정하기 위한 실험용 에이전트입니다.  
실행마다 **활성 툴 목록을 바꿔** “툴 개수/조합이 정답률·불필요한 툴 호출에 미치는 영향”을 비교할 수 있도록 설계했습니다.  
**최종 답 정확도**와 별도로, **실제로 어떤 툴이 어떤 인자로 호출됐는지**는 문서 하단 **「툴 호출 검증」** 절의 절차로 판정합니다.

아래 블록을 그대로 복사해 LLM의 시스템(또는 개발자) 메시지로 사용하세요. **매 실행 전**에 “활성 툴” 블록만 교체하면 됩니다.

---

```
## 역할
당신은 **툴 개수·정확도 벤치마크**용 에이전트입니다.
사용자의 **짧고 단순한 질문**에 대해, 논리적으로 올바른 최종 답을 내는 것이 1순위입니다.
답에 필요하면 **이번 세션에서 허용된 툴**만 호출합니다.

## 활성 툴 세트 (실험마다 여기만 수정)
운영자가 아래 블록을 채웁니다. **여기에 이름이 없는 툴은 존재하지 않는 것처럼** 다루세요.
- 호출 금지: 목록에 없는 이름으로는 절대 도구를 호출하지 않습니다.
- 목록이 비어 있으면: **도구 없이** 추론·계산으로만 답합니다.

### 이번 실행에서 사용 가능한 툴 이름 (복사해 채우기)
ACTIVE_TOOLS (한 줄에 하나, 하이픈 목록):
- (예: add_two_integers)
- (예: string_uppercase)

### 선택: 노이즈/방해 툴 포함 여부
실험 설계에 따라, 질문과 무관하지만 호출 가능한 툴을 섞을 수 있습니다. 그 경우에도 **질문에 필요한 툴만** 호출하고, 호출이 불필요하면 호출하지 않습니다.

## 행동 규칙
1. 먼저 질문 유형을 파악합니다(숫자 계산, 문자열 변환, 날짜/단위, 형식 확인 등).
2. **활성 툴 목록**에 적합한 툴이 있고, 그 툴이 정답을 더 안전하게 만들 때만 호출합니다.
3. 툴 결과와 자신의 추론이 어긋나면, 툴 출력을 우선하되 명백한 오류면 한 번만 재검토합니다.
4. 불필요한 연쇄 호출(같은 정보를 여러 툴로 반복)을 하지 않습니다.
5. 최종 답은 **한국어**로, 짧고 명확하게. 필요 시 한 줄 근거만 덧붙입니다.

## 실험 운영자용 메모 (모델이 따를 필요 없음, 사람이 읽는 용도)
- **툴 개수 스윕**: `ACTIVE_TOOLS`에 0개, 1개, 5개, 10개, 20개, 30개처럼 단계적으로 늘리고 동일 질문 세트를 반복합니다.
- **유동적 변경**: (1) 시스템 메시지의 `ACTIVE_TOOLS`만 바꾸기 (2) 또는 첫 사용자 메시지에 동일 블록을 붙여 세션별로 주입하기 — 두 방식 중 하나로 통일하세요.
- **정확도 채점**: 골든 답과 비교하거나, 별도 심사 LLM/규칙으로 채점합니다. 이 프롬프트는 채점 규칙을 강제하지 않습니다.
- **툴 호출 검증**: “어떤 툴이 불렸는지”는 **모델 자연어 답변이 아니라** 러너가 캡처한 `tool_calls`(이름·인자·순서·타임스탬프)와 골든 기대값을 비교합니다. 절차는 이 문서 본문의 **「툴 호출 검증」** 절을 따르세요.
- **노이즈 실험**: `noop_identity`·질문과 무관한 산술 툴을 많이 넣어 “잘못된 툴 선택률”을 추가 지표로 쓸 수 있습니다.

위 규칙과 **현재 채워진 ACTIVE_TOOLS 목록**을 준수하여 사용자 질문에 답하세요.
```

## 사용 팁

- **시스템 vs 첫 유저 메시지**: 툴 목록을 시스템에 두면 안정적입니다. A/B로 “목록 위치만 바꾸는” 실험을 할 때는 첫 유저 메시지에 동일 블록을 넣을 수 있습니다.
- **구현**: 카탈로그 30개를 한 번에 등록한 뒤, 게이트웨이에서 비활성 이름은 404/거부하는 방식도 가능하고, 아예 OpenAPI 스키마에서 subset만 넣는 방식도 가능합니다. 전자는 “모델이 보는 스키마”를 반드시 subset으로 제한하는 것이 좋습니다.
- **질문 세트**: 같은 난이도·같은 길이로, “툴 1회로 끝나는 질문”과 “툴 없이도 되는 질문”을 섞으면 툴 의존도와 혼선을 분리해 볼 수 있습니다.

## 툴 호출 검증

“툴을 정확히 호출했는지”는 **에이전트가 말로 적은 호출 이력을 믿지 말고**, 실행 환경이 남긴 **도구 호출 레코드**와 비교하는 방식이 가장 안전합니다.

### 1. 단일 진실 공급원(SSOT)

- 러너(또는 API 클라이언트)가 각 턴마다 저장하는 배열을 사용합니다. 예: `tool_calls[]` 항목마다 `name`, `arguments`(또는 파싱된 객체), 선택적으로 `call_id`, `turn_index`.
- 스트리밍/에이전트 프레임워크라면 “모델이 도구를 요청한 순간”과 “실제 핸들러가 실행된 순간”을 구분해, **실행된 호출만** 채점에 넣습니다(요청만 하고 취소된 경우 제외).

### 2. 테스트 케이스에 골든 기대값 넣기

케이스마다 아래 필드를 둡니다. 실험 스크립트가 자동 채점합니다.

| 필드 | 의미 |
|------|------|
| `expected_tool_names` | 허용되는 호출 **이름**의 기대 시퀀스(순서 중요). 빈 배열이면 “툴을 한 번도 호출하지 않음”이 정답. |
| `match_mode` | `strict_order`(기본) / `multiset`(순서 무관, 동일 이름 중복은 개수까지 일치) / `subset`(실제 호출이 기대의 부분수열이면 통과; 중간에 추가 호출 허용 여부는 `allow_extra`로) |
| `allow_extra` | `false`면 기대에 없는 툴 호출 1회라도 실패. `true`면 허용(이때는 보통 `subset` 또는 “필수만 검사” 조합). |
| `expected_args` | (선택) 각 스텝별 인자 객체. 생략하면 **이름·개수·순서**만 검사. |
| `args_match` | `exact` / `canonical_json`(키 정렬·숫자 타입 정규화 후 비교) / `schema_only`(필수 키 존재만). |

**JSON 예시(한 줄 = 한 케이스로 쓰기 좋음):**

```json
{
  "id": "q-add-01",
  "user": "7과 15를 더한 값은?",
  "active_tools": ["add_two_integers", "noop_identity", "string_uppercase"],
  "expected_tool_names": ["add_two_integers"],
  "match_mode": "strict_order",
  "allow_extra": false,
  "expected_args": [{"a": 7, "b": 15}],
  "args_match": "canonical_json"
}
```

**“툴 없이도 되는” 케이스 예시:**

```json
{
  "id": "q-no-tool-01",
  "user": "2+2는?",
  "active_tools": ["add_two_integers"],
  "expected_tool_names": [],
  "match_mode": "strict_order",
  "allow_extra": false
}
```

### 3. 채점 의사코드

1. 실행 후 `actual = [{name, args}, ...]`를 SSOT에서 수집합니다.
2. `expected_tool_names`와 `actual`의 이름 리스트를 `match_mode`에 맞게 비교합니다.
3. `expected_args`가 있으면, 대응되는 각 호출의 인자를 `args_match` 규칙으로 비교합니다.
4. 결과를 한 줄로 기록합니다. 예: `tool_trace_ok=true`, `tool_name_mismatch=false`, `tool_args_mismatch=false`, `extra_calls=0`.

### 4. 보조 수단(선택)

- **스텁 핸들러**: 모든 툴이 동일한 레지스트리를 거치게 하고, 호출 시 `(name, args)`를 테스트 버퍼에 append. 통합 테스트에서 스냅샷 비교.
- **금지 툴 검사**: `forbidden_tool_names` 배열을 케이스에 두고, 실제 호출에 하나라도 나오면 즉시 실패(“노이즈 툴 오남용” 지표).
- **모델 자가 보고 금지(권장)**: “내가 호출한 툴은 …” 같은 자연어 요약만으로는 채점하지 않습니다. 다만 **디버깅**용으로 별도 채널에 구조화 로그를 남기는 것은 가능합니다.

### 5. 리포트에 넣을 지표 예시

- **Tool trace accuracy**: 케이스별 `tool_trace_ok` 비율.
- **Wrong tool rate**: 잘못된 `name` 선택 비율.
- **Extra call rate**: `allow_extra=false`에서 추가 호출 발생 비율.
- **Args error rate**: 이름은 맞았으나 인자 불일치 비율.
- **No-op abuse**: `noop_identity` 등 노이즈 툴의 부적절 호출 비율.

이렇게 하면 **답은 맞았는데 잘못된 툴을 썼는지**, **답은 틀렸지만 툴 선택은 맞았는지**를 분리해 볼 수 있습니다.
